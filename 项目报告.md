## 一、题目需求描述

**选题背景与意义**：
美食内容平台需高效处理用户投稿与人工审核流程，传统方案常导致内容发布延迟。本系统通过 Flask 动态审核 + MkDocs 静态归档 实现“投稿-审核-自动发布”闭环，提升内容运营效率。

**核心功能需求（必做）**：

1.  **用户投稿**：支持图文投稿（标题、描述、分类、图片），数据存入 SQLite；
2.  **管理员审核**：审核待审内容（通过/拒绝），状态更新至数据库；
3.  **自动归档发布**：审核通过后，自动将内容转为 Markdown 文件并触发 MkDocs 重建，生成静态页面发布。

**拓展功能需求（选做/加分项）**：

1.  实现待审投稿的优先级排序（如按时间）；

## 二、核心数据结构与算法设计

1.  **核心数据结构（如链式二叉树、邻接表图、哈希表、并查集等）**：
    - **数据库表结构**：使用SQLite数据库，包含users表（用户信息）和food_submissions表（投稿信息），采用关系型数据模型
    - **数据传输结构**：使用sqlite3.Row对象进行数据传输，支持字典式访问
    - **文件系统结构**：采用目录层次结构管理静态文件和Markdown文档
2.  **核心算法策略（如递归/非递归遍历、Dijkstra算法、快速排序、KMP模式匹配等）**：
    - **数据库查询**：使用SQL语句进行数据操作，如SELECT、INSERT、UPDATE
    - **文件名清理**：使用正则表达式清理非法字符
    - **Markdown生成**：字符串拼接生成标准化Markdown格式
    - **MkDocs重建**：子进程调用实现静态站点重建
    - **审核队列排序**：按提交时间降序排序实现优先级管理
3.  **算法的时间复杂度预估（考虑最坏情况）**：
    - **数据库查询**：O(1)（有索引）或O(n)（无索引）
    - **文件名清理**：O(n)，其中n是文件名长度
    - **Markdown生成**：O(1)，固定操作
    - **MkDocs重建**：O(1)，外部命令调用
    - **审核队列排序**：O(n log n)，数据库排序
4.  **算法的空间复杂度预估**：
    - **数据库存储**：O(n)，其中n是数据量
    - **内存使用**：O(1)，常量级
5.  **预期的数据规模**：
    - 单日投稿量：≤ 100 条
    - 待审队列长度：≤ 50 条

## 五、可行性分析与难点预估

**技术难点/可能遇到的困难**：

1.  MkDocs 静态生成与 Flask 动态审核的同步（MkDocs 无法实时更新）
2.  审核队列的高效管理
3.  文件名规范化处理（防止非法字符导致文件创建失败）
4.  静态目录结构的自动维护

**解决方案的初步思考**：

1.  **MkDocs 同步问题**：实现审核通过后自动调用 `mkdocs build` 命令重建静态站点，确保内容及时更新
2.  **审核队列管理**：使用 SQL 语句 `ORDER BY submit_time DESC` 实现按时间降序排序，优先处理最新投稿
3.  **文件名规范化**：使用正则表达式清理非法字符，将空格替换为下划线，确保文件名合法
4.  **目录结构维护**：审核通过后自动创建对应分类目录，并更新 sanshui_out.md 文件，保持目录结构的一致性
